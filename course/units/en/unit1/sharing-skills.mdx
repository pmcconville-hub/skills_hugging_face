# Sharing Skills

You've written a skill and used skills from the Hub. Now let's close the loop: how do you share your skills with others so they can benefit from your expertise?

## Why Share Skills?

Sharing skills multiplies their impact. A skill you write to solve your own workflow can help hundreds of other developers facing the same challenge. The Hugging Face ecosystem thrives on this kind of community contribution — just as shared models and datasets advance the field, shared skills raise the bar for what coding agents can do.

## Publishing to the HF Skills Repository

The most direct way to share a skill is by contributing to the [huggingface/skills](https://github.com/huggingface/skills) repository.

### Step 1: Prepare Your Skill

Make sure your skill directory follows the standard structure:

```
your-skill/
├── SKILL.md              # Required: main instruction file
├── scripts/              # Optional: helper scripts
├── references/           # Optional: supplementary docs
├── templates/            # Optional: starter templates
└── examples/             # Optional: worked examples
```

Your `SKILL.md` must have valid frontmatter:

```yaml
---
name: your-skill-name
description: A clear, concise description of what the skill does
license: MIT
metadata:
  owner: your-username
  tags: [relevant, tags, here]
---
```

### Step 2: Validate Your Skill

Before submitting, check that:

- The `SKILL.md` frontmatter has `name` and `description` fields
- All relative paths in the SKILL.md point to files that exist
- Scripts are executable and include necessary imports
- The skill has been tested with at least one coding agent
- The instructions are clear enough for an agent to follow without ambiguity

Use the Agent Skills reference validator before opening a PR:

```bash
npx @agentskills/skills-ref validate ./your-skill
```

### Step 3: Submit a Pull Request

Fork the repository, add your skill, and open a PR:

```bash
# Fork and clone
git clone https://github.com/your-username/skills.git
cd skills

# Add your skill
cp -r /path/to/your-skill/ ./your-skill/

# Commit and push
git add your-skill/
git commit -m "Add your-skill: brief description"
git push origin main

# Open a PR on GitHub
```

Include in your PR description:

- What the skill does
- Which agent(s) you tested it with
- Example prompts that demonstrate the skill in action

## Using upskill for Skill Generation

The [huggingface/upskill](https://github.com/huggingface/upskill) tool can help you generate and evaluate skills systematically. It uses a "teacher-student" approach: a powerful model generates skill instructions that cheaper, faster models can then use to perform complex tasks.

### Generate a Skill

```bash
upskill generate --task "Create CUDA kernels for transformer operations"
```

This produces a `SKILL.md` and supporting files based on the task description.

### Evaluate a Skill

```bash
upskill eval --skill ./your-skill/ --model claude-sonnet
```

This runs test cases against your skill and measures:

- **Success rate**: How often the agent completes the task correctly
- **Token efficiency**: How many tokens the agent uses (fewer is better)
- **Baseline comparison**: Performance with and without the skill

### Benchmark Across Models

```bash
upskill eval --skill ./your-skill/ --model claude-sonnet --model gpt-4o --model gemini-pro
```

This shows you how your skill performs across different coding agents, helping you identify instructions that are too specific to one model.

## Sharing via the Hub

You can also share skills as repositories on the Hugging Face Hub:

```bash
# Create a repo
hf repo create your-skill --type model

# Upload your skill directory
hf upload your-skill ./your-skill
```

Others can then install your skill directly:

```bash
# Clone from the Hub
git clone https://huggingface.co/your-username/your-skill
```

## Best Practices for Shared Skills

When sharing skills with the community, keep these guidelines in mind:

**Test with multiple agents.** A skill that works perfectly with Claude Code might confuse Codex or Gemini CLI. Test with at least two agents and adjust your instructions for clarity.

**Include examples.** Add a `examples/` directory with worked examples showing the skill in action. This helps both human readers and agents understand the expected workflow.

**Version your skills deliberately.** Track versions in repository tags/releases or structured `metadata` fields, and bump major versions for breaking workflow changes.

**Document limitations.** Be upfront about what the skill can and can't do. If it only works with certain model sizes, GPU types, or data formats, say so in the SKILL.md.

**Keep it maintained.** Skills reference external tools and APIs that change over time. Check your skills periodically and update them when dependencies evolve.
